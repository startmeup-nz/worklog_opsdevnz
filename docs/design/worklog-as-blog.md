# Design Decision: Worklog as Blog via Zensical

**Status:** Decision — Blog integration deferred  
**Created:** 2026-04-28  
**Spec Reference:** FR-1 (entry creation), FR-5 (listing)

---

## Problem Statement

Worklog entries are structurally similar to blog posts: dated, frontmatter-tagged
Markdown files with an author and optional tags. Other teams using our worklog
pattern have asked whether we should integrate with Zensical's blog feature to
get chronological navigation, tag filtering, and RSS feeds.

The question: **Should worklog-opsdevnz integrate with Zensical's blog system,
or remain a standalone file-management tool?**

---

## Options Considered

### Option A: Integrate with Zensical Blog

Treat worklogs as a Zensical blog collection. Leverage Zensical's built-in
chronological ordering, tag pages, and RSS generation.

**Pros:**
- Chronological navigation out of the box
- Tag filtering and author pages
- RSS feed for worklog entries
- Consistent with how blogs work in Zensical

**Cons:**
- Zensical auto-navigation sorts by filename/folder alphabetically, not by
  `date` frontmatter — worklogs need date ordering
- Flat structure doesn't scale (100+ entries in one directory)
- Year or year-month structure breaks chronological display in auto-nav
- Zensical has no documented mechanism for custom section labels with
  auto-navigation (we already hit this with `dns-as-code` → "Dns As Code")
- Would couple worklog-opsdevnz to Zensical, reducing its usefulness as a
  standalone tool for non-Zensical projects
- Requires investigation into Zensical plugin architecture or feature request
- **Time cost unknown** — we have end-of-month beancount reconciliation and
  ledger security review taking priority

### Option B: Standalone Tool + Auto-Generated Index

Keep worklog-opsdevnz as a standalone CLI. Add an `index` subcommand that
generates a reverse-chronological `index.md` from existing entries.

**Pros:**
- No dependency on Zensical — works with any static site generator
- `index.md` provides chronological listing regardless of directory structure
- Can be enhanced later with Zensical integration if/when needed
- Lower time cost — we already have the prototype, just need to package it
- Other teams can adopt the tool without adopting Zensical

**Cons:**
- No RSS feed (would need separate tool or Zensical blog integration later)
- No tag filtering pages (would need Zensical or manual index generation)
- Navigation in Zensical sidebar still alphabetical, not chronological

### Option C: Hybrid — Standalone + Zensical-Compatible Output

Build the standalone tool (Option B) but ensure the frontmatter and file
structure are Zensical-blog-compatible. When a project wants blog features,
they configure Zensical to treat the worklog directory as a blog collection.

**Pros:**
- Best of both worlds: standalone now, blog-ready later
- Zero coupling to Zensical in the core tool
- Migration path is trivial: just add Zensical config

**Cons:**
- Requires understanding Zensical's blog collection format
- Still no chronological sidebar navigation without Zensical changes

---

## Decision: Option C (Hybrid)

**We will build worklog-opsdevnz as a standalone CLI tool (Option B) with
Zensical-compatible frontmatter and structure (Option C). Blog integration
is explicitly out of scope for 0.0.1 and 0.1.0.**

### Rationale

1. **Time constraint** — End-of-month beancount reconciliation and ledger
   security review are the sprint priorities. Blog integration research is
   an unknown time sink.

2. **Standalone value first** — The core problem is standardising the ad-hoc
   worklog scripts across three projects. That problem is solved by packaging
   the existing prototype. Blog features are a nice-to-have.

3. **No coupling** — Other teams asking for this tool may not use Zensical.
   Keeping it standalone maximises adoption.

4. **Migration path is trivial** — If we later decide to integrate with
   Zensical blog, the frontmatter format is already compatible. The only
   change would be adding Zensical-specific config to the consuming project.

### What This Means for 0.0.1 / 0.1.0

- **In scope:** Create, list, init commands. Config-driven templates. Three
  structure modes. Editor integration.
- **Out of scope:** Zensical blog integration, RSS feeds, tag pages,
  chronological sidebar navigation.
- **Future consideration:** `worklog-opsdevnz generate-index` subcommand
  that produces a reverse-chronological `index.md` with links, dates, and
  tags. This would solve the navigation problem without Zensical coupling.

### Recommended Directory Structure

For projects using Zensical, we recommend `year-month` structure:

```
docs/worklog/
├── index.md              # Auto-generated by worklog-opsdevnz (future)
├── 2026/
│   ├── 04/
│   │   ├── 2026-04-28-module-scaffolding.md
│   │   └── 2026-04-27-python-3-14-maintenance.md
│   └── 03/
│       └── 2026-03-28-worklog.md
└── 2025/
    └── 12/
        └── 2025-12-15-year-end-review.md
```

This keeps directories manageable (10-30 entries each) and enables archive
browsing. The `index.md` (auto-generated) provides the chronological listing
that Zensical auto-navigation cannot.

### Zensical Limitations to Track

| Limitation | Impact | Workaround |
|-----------|--------|------------|
| Auto-nav sorts alphabetically, not by date | Sidebar shows entries out of order | Auto-generated `index.md` with correct ordering |
| No custom section labels in auto-nav | `dns-as-code` → "Dns As Code" | Use explicit nav for section labels, auto for entries |
| No glob patterns in nav config | Can't auto-include all worklog files | Auto-discovery mode (already supported) |

## UX Requirements (2026-05-22)

From opsdev: the worklog should feel like a blog, not a file listing.

1. **No sidebar clutter** — individual worklog entries must not appear as links in
   the Zensical navigation sidebar. Only the worklog index/landing page should
   appear.

2. **Browse by date** — entries should be navigable chronologically (reverse
   chronological listing), with clear date grouping (year → month).

3. **Tag browsing** — tags set in frontmatter should be filterable, allowing the
   reader to see all entries tagged `opsdev` or `ramp` or `beancount`, etc.

4. **Auto-generated index** — a reverse-chronological `index.md` listing all
   entries with dates, titles, and tags. This is the primary navigation surface.

5. **Zensical blog integration (future)** — if Zensical adds tag-filtering or
   chronological blog features, adopt them. Until then, the auto-generated index
   page is the navigation hub.

### How Zensical handles this

Zensical auto-navigation (the sidebar) picks up every `.md` file in the docs
directory unless explicitly excluded. To avoid sidebar clutter from individual
worklog entries, we need to explore:

- Zensical's `nav` exclusion mechanism
- Placing worklog entries outside the docs_dir
- Using a plugin or custom template to hide them

This is tracked as a design investigation item.
